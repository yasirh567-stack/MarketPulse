# Backtesting Methodology

**All results produced by this engine are hypothetical and based on historical or
synthetic demo data. Past performance — simulated or real — does not guarantee
future results. Nothing here is financial advice.**

## Strategy

A configurable long-only threshold strategy:

- **Enter** at the next trading day's **open** when, using only information known
  as of today's close, the model's out-of-sample probability of "up" exceeds
  `prob_threshold` **and** trailing 5-day sentiment exceeds `sentiment_threshold`.
- **Exit** at the next trading day's open when any of:
  - `holding_period_days` have elapsed since entry,
  - the current day's probability has fallen below `exit_prob_threshold`,
  - the backtest period has ended (forced close-out — a position is never left open
    past the simulated period).

No shorting, no leverage, one position at a time, 100% of available capital per trade.

## No-look-ahead guarantee

The probability series driving trade decisions is **not** generated by a single
model fit on the whole history. It reuses the exact same expanding-window
walk-forward folds as the research pipeline (`app/backtesting/signal_generator.py`):
for each fold, a model is fit **only** on data strictly before that fold's test
block, then scores that block. Concatenating every fold's scores gives a full
out-of-sample probability series where no day's signal was produced by a model that
had seen that day, or any later day, during training.

Trade **execution** further avoids look-ahead by filling at the *next* day's open
rather than the signal day's own close — a decision made using information known as
of day T's close cannot be executed at a price from day T itself.

Days before the first fold's test block (the initial expanding-window warm-up, ~40
trading days) have no out-of-sample signal and are excluded from the backtest.

## Costs

Both `transaction_cost_bps` and `slippage_bps` are applied on **every fill** (entry
and exit), each as a basis-point adjustment to the fill price against the trader
(buys pay up, sells receive less). Defaults: 10 bps transaction cost, 5 bps
slippage — configurable per run, including zero for a frictionless baseline.

## Metrics

| Metric | Definition |
|---|---|
| Total return | Final equity / initial capital − 1 |
| Annualized return | Total return compounded to a 252-trading-day year |
| Annualized volatility | Std. dev. of daily returns × √252 |
| Sharpe ratio | Mean daily return / std. dev. of daily returns × √252 |
| Sortino ratio | Like Sharpe, but the denominator only counts downside deviation |
| Max drawdown | Largest peak-to-trough decline in the equity curve |
| Win rate | Share of closed trades with positive return |
| Number of trades / turnover | Completed round-trips / total fills (buy+sell) |
| Avg. trade return | Mean percentage return per completed trade |
| Exposure | Share of the backtest period spent holding a position |

All computed directly from the simulated equity curve and trade log — never
hand-adjusted.

## Benchmark

Buy-and-hold over the exact same date range and starting capital, using the same
`open_prices`/`close_prices` series, is always computed and charted alongside the
strategy for comparison.

## Train / validation / test framing

The engine supports an explicit `start_date`/`end_date` window on the request,
letting a user restrict the simulation to a specific test period distinct from
whatever earlier data warmed up the walk-forward folds. If the requested window is
narrower than 10 out-of-sample trading days, the response includes a warning that
the sample is too small to be meaningful, rather than silently returning polished-
looking metrics on statistically thin data.

## Known simplifications

- Fills occur exactly at the next bar's open plus/minus costs — no partial fills,
  no market-impact modeling, no bid/ask spread beyond the slippage parameter.
- Position sizing is always "all in, one position" — no portfolio-level risk sizing.
- The signal-generation folds are fixed at 5 (matching the research pipeline);
  very short histories may produce very few, coarse folds.
- Demo tickers' underlying price series are synthetic (see `docs/data-sources.md`),
  so demo backtest results describe a synthetic market, not a real one.
- The confidence-threshold sweep (below) is stateless — never persisted as a
  `BacktestRun` row. The one expensive step (`build_oos_signal_series`, i.e. the
  walk-forward model fitting) runs exactly once per sweep request regardless of
  how many thresholds are compared, so unlike a single `POST /backtests` call
  there's no caching benefit to storing results, and it mirrors the existing
  stateless `/instruments/screener` endpoint rather than the persisted backtest
  endpoint.

## Confidence-threshold sweep

`POST /api/v1/backtests/threshold-sweep` runs the identical strategy at several
`prob_threshold` values against the *same* out-of-sample signal (built once via
`build_oos_signal_series`, then replayed through the cheap `run_backtest`
simulation loop per threshold via `dataclasses.replace`) — this answers whether
only trading when the model is more confident actually improves risk-adjusted
returns, or just means fewer trades with no edge.

**Testable invariant:** holding every other config field fixed, raising
`prob_threshold` can only make the entry condition *more* restrictive, so
`num_trades` must be non-increasing as the threshold rises. Regression-tested in
`backend/tests/test_backtesting.py::test_num_trades_non_increasing_as_prob_threshold_rises`.

**What running it against real (live-mode) data actually showed:** across the
bundled tickers, a 0.50→0.70 threshold sweep rarely changes anything — the
model's calibrated out-of-sample probabilities cluster near the extremes rather
than spreading smoothly across the range, so most tickers show identical trades
at every threshold in that band (e.g. AAPL: 2 trades, Sharpe 0.13, unchanged
across 0.50–0.80). Where the threshold *does* bind, it isn't a clean "fewer but
better" story: NVDA drops from 1 profitable trade (+4.4% return) to 0 trades the
moment the threshold crosses its one high-confidence signal, and GME's Sharpe
actually gets *worse* (−0.18 → −0.64) when a 0.90 threshold filters down to only
its highest-confidence trades. **Honest conclusion:** for this model, feature set,
and history length, a stricter confidence gate does not reliably buy better
risk-adjusted performance — it mostly acts as a blunt on/off filter around
whatever threshold happens to sit near the model's few confident signals, not a
continuous quality dial. This is a real (if modest) finding, not a demo-data
artifact, and is reported here rather than hidden because a null result is still
a result.

## Tests

`backend/tests/test_backtesting.py` directly asserts: trades execute and exit on
schedule; non-zero transaction costs strictly reduce final equity vs. a zero-cost
run; no trade fires when probability never crosses the threshold; drawdown is
never positive and is exactly zero at new equity highs; entry fills use the next
day's open (not the signal day's close); and a still-open position is force-closed
at the end of the period.
