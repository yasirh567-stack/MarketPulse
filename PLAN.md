## Guiding constraints recap

- Must run for **free**, no paid APIs/cloud required.
- Demo mode must let a recruiter exercise every page with **zero** API keys.
- Every data point is labeled: `live | delayed | historical | cached | demo`.
- No fabricated data, no financial-advice framing, no guaranteed-return claims.
- Time-series ML/backtests use walk-forward validation — never random shuffling.

## Environment notes (discovered during setup)

- Host had Python 3.14 (too new for reliable numpy/pandas/sklearn wheels) and no
  Node.js. Homebrew's `node`/`python@3.11` install required a from-source build of
  several dependencies (slow on this host's macOS version) — switched to installing
  Python 3.11 and Node 20 via a `conda` environment instead, which had prebuilt
  packages available. The backend uses a `.venv` created from that conda Python
  3.11 (`backend/.venv`); the frontend uses that conda environment's Node/npm.
- No Docker installed on host — Docker/Compose files are authored per spec (for
  grading/CI/other machines) but local verification of the app itself was done via
  native `venv` + `npm run dev`/`vite build`, not `docker compose up`. Called out
  explicitly in the final report rather than silently assumed.
- Not a paid-API environment: yfinance (free, unofficial Yahoo endpoints) is the
  live/delayed market data source; Google News RSS (no key required) is the news
  source; Reddit integration is optional/off by default since it needs OAuth
  credentials. FinBERT is optional (Transformers+Torch are heavy) and downgrades to
  VADER automatically.

### Bug caught and fixed during manual verification
The first end-to-end test of `/predictions/{ticker}` returned a 97%-confident "up"
prediction while the walk-forward accuracy was ~49.5% (a coin flip) — a real bug, not
a demo-data artifact. Root cause: the final model's probability calibrator was being
fit on the same data the model was trained on, so a gradient-boosted model that had
memorized its own training rows produced near-0/1 "calibrated" probabilities at
inference. Fixed by training the final model on the first 80% of history and fitting
the calibrator only on the later, held-out 20% — same principle as the walk-forward
folds, applied to the production model too. Regression-tested in
`test_ml_pipeline.py::test_calibrator_not_fit_on_same_data_it_trained_on`.

## Explicit simplifications (documented, not hidden)
- Reddit/social data ships as an optional, off-by-default provider (needs user-supplied
  OAuth credentials) — the platform is fully functional without it via news sentiment alone.
- SHAP is optional-enhancement only; permutation importance is the default explainability
  path everywhere so the app works without SHAP's heavier native deps.
- Redis is optional; default cache is a SQLite/in-memory TTL cache.
- "Real-time" means polling-interval push (configurable, default suited to free data
  rate limits), never sub-second ticks — this is disclosed in the UI and docs.
- Docker/Compose are authored to spec but not exercised on this host (no local Docker
  engine); native venv/npm run is the verified local path.
- The production ML model trains on the first 80% of available history, holding out
  the most recent ~20% purely for calibration (see "Bug caught" note above) — a
  deliberate recency-vs-honesty tradeoff, documented in `docs/model-card.md`.
- npm audit flags a moderate esbuild advisory that only affects the local Vite dev
  server (not the production build/runtime); not force-upgraded to avoid an
  unplanned breaking change to Vite mid-project. See `docs/deployment.md`.

### Bug caught and fixed during this verification
Manually verifying a brand-new live ticker (NFLX, no bundled fixture) end-to-end
in a real browser surfaced a genuine pre-existing race condition: `_persist_bars`
(`backend/app/services/market_service.py`) checked for existing `(ticker,
interval, ts)` rows and then inserted the missing ones in a separate step — two
overlapping requests for the same ticker (different `period_days` windows, or a
React remount re-firing the same query) could both pass the "does it exist"
check before either committed, then both try to insert the same row, raising a
`sqlite3.IntegrityError` on the unique index and surfacing to the browser as a
500 (which in turn looked like a CORS failure, since the crashed response had
no CORS headers). Fixed by replacing the check-then-insert with a single atomic
`INSERT ... ON CONFLICT DO NOTHING` upsert (dialect-aware: SQLite or Postgres),
which makes the race structurally impossible rather than just less likely.

## Post-completion enhancement: model leaderboard + confidence-threshold sweep
Closes the README's former "Future improvements" item ("a model leaderboard /
drift view"). Two pieces:

- **`GET /api/v1/leaderboard`** (`backend/app/api/v1/leaderboard.py`,
  `backend/app/services/leaderboard_service.py`) — compares walk-forward metrics
  across several tickers in one call. No new training code path or DB table:
  it reuses `app.ml.service.get_or_train`'s existing staleness/auto-train logic
  per ticker and reads the already-persisted, append-only `model_runs` table.
  Each row reports `beats_baseline` (combined balanced accuracy vs. the stronger
  of the majority-class/previous-direction baselines). A ticker that fails is
  marked on its own row (`status: "insufficient_data"` or `"error"`, with a
  `note`) rather than failing the whole request — this required catching both
  `InsufficientDataError` *and* `ProviderUnavailableError`, since an unresolvable
  ticker fails at the market-data layer with the latter, not the former, and an
  earlier draft that only caught `InsufficientDataError` would have let one bad
  ticker 503 the entire leaderboard.
- **`POST /api/v1/backtests/threshold-sweep`** (`backend/app/backtesting/service.py::run_threshold_sweep`)
  — runs the identical backtest strategy at several `prob_threshold` values
  against the *same* out-of-sample signal (built once via
  `build_oos_signal_series`, then replayed through the cheap `run_backtest` loop
  per threshold), to test whether trading only on high-confidence signals
  actually improves risk-adjusted returns. Deliberately stateless — no new
  `BacktestRun`-style table — since the one expensive step runs exactly once per
  request regardless of how many thresholds are compared. Frontend: a "Compare
  confidence thresholds" button on the existing Backtesting page (not a new
  page), since it's config-driven off the same strategy parameters.

### Bugs caught and fixed while verifying this against real data
Running the new sweep endpoint against AAPL (live mode, real yfinance data)
immediately 500'd — and so did the *existing*, unmodified single `POST
/backtests` endpoint on the same ticker, proving this was a pre-existing bug,
not something the new code introduced. Root cause: `load_bars_frame`
(`backend/app/ml/service.py`) built a `pandas.DatetimeIndex` directly from raw
`yfinance` bar timestamps, which are localized to the exchange's own timezone
(`America/New_York`). A 400-day window almost always crosses at least one DST
transition, so the list mixes two different fixed UTC offsets (EDT/EST) —
`pandas.DatetimeIndex` refuses to build an index from mixed-offset tz-aware
datetimes without an explicit UTC normalization first. This would have broken
predictions, backtests, and now the leaderboard/sweep for *any* real ticker with
more than ~6 months of live history — invisible in demo mode (synthetic fixtures
carry no real timezone data) and easy to miss with short backtest windows.
Fixed by normalizing every bar timestamp to UTC before constructing the index.

### What the sweep actually showed, run for real
Across the bundled tickers (AAPL, NVDA, TSLA, MSFT, GME), a 0.50–0.70 threshold
sweep mostly changed nothing — the model's calibrated out-of-sample
probabilities cluster near the extremes rather than spreading smoothly, so most
tickers took the identical trades at every threshold in that band. Where the
threshold did bind, it wasn't a clean "fewer but better" story: NVDA's one
profitable trade (+4.4%) disappeared entirely past 0.65 rather than being
replaced by a higher-quality one, and GME's Sharpe ratio got *worse* (−0.18 →
−0.64) once a 0.90 threshold filtered down to only its highest-confidence
trades. Honest conclusion, written up in `docs/backtesting-methodology.md`: for
this model/feature set/history length, a stricter confidence gate does not
reliably buy better risk-adjusted performance. Reported as-is rather than
tuned or cherry-picked to look like a positive result.
